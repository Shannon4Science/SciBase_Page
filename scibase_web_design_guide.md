# SciBase 官网原型设计说明文档

## 1. 项目定位

**SciBase** 是面向大模型时代的 AI-Ready 学术知识基础设施。它不是传统文献检索系统，而是一个可计算、可理解、可被 AI 直接使用的知识聚合体，覆盖公开学术文献、图书、专利等多类型知识资产。

核心定位：

> 从“文献检索”升级为“知识基座”：不仅让人类检索，更让 Research Agent、Prior-Art Agent、Technical Intelligence Agent 获得可验证的证据层。

核心特征：

- **全面性**：跨学科、多类型、多来源、多语种。
- **实时性**：持续增量采集，可追溯更新。
- **AI-Ready**：面向模型训练、RAG 推理、Agent 知识调用设计。

---

## 2. 页面目标

该页面是 SciBase 的产品介绍首页，用于向外部用户解释：

1. SciBase 是什么。
2. SciBase 的数据规模和覆盖能力。
3. SciBase 如何服务人类用户与 Agent。
4. SciBase 背后的 Harvest 数据采集与治理架构。
5. 用户如何开始使用：API、在线体验、联系我们。

页面不应设计成传统数据库门户，而应呈现为一个**学术感、基础设施感、数据可信感**兼具的产品介绍页。

## 3. 目标用户

### 3.1 2B 用户，主要目标

包括：

- Auto-Research / Deep Research 产品团队。
- 企业数据挖掘服务团队。
- 投研、咨询、药研、知识产权、技术情报团队。
- 大模型公司、RAG 平台、企业知识库厂商。

他们关注：

- 数据规模。
- 数据来源可信度。
- 是否有全文与证据片段。
- 是否适合 Agent 调用。
- 是否可通过 API 集成。
- 是否有数据治理、版权、质量和追溯能力。

### 3.2 2C 用户，次要目标

包括科研人员、学生、技术研究者、普通检索用户。

他们关注：

- 是否可以检索论文、专利、图书。
- 是否有良好的在线体验。
- 是否能快速看到权威结果。

## 5. 视觉风格

### 5.1 总体风格

采用**极简学术风**：

- 米白背景。
- 细线边框。
- 大字号衬线标题。
- 少量棕色强调。
- 避免高饱和科技蓝紫渐变。
- 不使用复杂 3D 或过度炫酷动效。

关键词：

```text
Minimal / Academic / Infrastructure / Trustworthy / Archival / Agent-ready
```

主色不超过三类：

| 用途 | 颜色建议 |

## 6.5 技术架构区：Harvest 系统动图

### 目的

展示 SciBase 的底层数据采集、治理和加工能力。

### 页面表现

该区域建议做成深色背景区，作为页面视觉重点。

主标题：

```text
Harvest 系统架构：从多源采集到 AI-Ready 知识对象
```

说明：

```text
此区域可设计为动图：数据流从左向右进入 Raw Lake，经标准化、解析、融合、治理后生成 Evidence Layer。
```

### 五段流水线

页面中展示五个主要阶段：

```text
Source Registry
→ Connector / Harvester
→ Raw Lake
→ Normalize & Parse
→ Knowledge & Evidence
```

#### 1. Source Registry

管理论文、专利、图书、标准、数据集等来源，记录授权、更新频率、字段映射和来源可信度。

#### 2. Connector / Harvester

通过 API、Dump、OAI-PMH、Feed、Crawl 等方式持续增量采集。

#### 3. Raw Lake

保存原始响应、PDF、HTML、XML、EPUB、图片和采集上下文。

#### 4. Normalize & Parse

统一 schema，解析章节、图表、公式、引用、claims 和全文结构。

#### 5. Knowledge & Evidence

形成 canonical objects、evidence spans、entity graph 与质量信号。

### 三个核心能力

在架构区底部展示三张卡片：

#### Canonicalization

多源记录归一为可信知识对象，解决同一论文、专利族、作者、机构和主题的重复问题。

#### Provenance

每个字段、证据片段和关系均保留来源、解析版本、时间戳与置信度，支持审计和复现。

#### Trust Signals

沉淀质量评分、版权边界、来源可信度、解析质量、撤稿风险和 freshness signal。

### 动效建议

- 数据点从左侧进入 pipeline。
- 在 Raw Lake 区域聚合。
- 经过 Normalize & Parse 时分裂成结构化块。
- 最终在 Knowledge & Evidence 区域形成网络节点。
- 动效低频、克制，避免“炫技”。

## 8. 开发注意事项

1. 当前原型是静态 HTML，可进一步转成 React / Next.js 组件。
2. 所有数字先作为静态展示，后续可接入配置文件或 CMS。
3. 概念图建议保持无文字，避免与页面正文重复。
4. 技术架构区未来可做 Canvas / CSS animation / Lottie 动效。
5. 指标图表当前为 CSS 条形图，后续可用 ECharts / Recharts。
6. 页面需要保留 anchor id，方便导航跳转。
7. 不建议过早增加复杂筛选和检索框，首页重点是解释定位与能力。
